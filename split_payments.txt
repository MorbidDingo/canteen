PLAN: PAYMENT SETTLEMENT & ACCOUNT MANAGEMENT SYSTEM
=====================================================

TL;DR
-----
Add a settlement account system where admins and management register bank accounts / UPI VPAs.
All canteen payments go to the platform's single Razorpay account with a visible 2% platform fee
surcharge. Daily automated payouts via RazorpayX send 98% to the admin's designated account.
Management has full authority: can view all accounts, block transactions, and override which
account receives canteen payments.


=======================================================
PHASE 1: DATABASE SCHEMA (new tables + migration)
=======================================================

1.1  New table: settlement_account
------------------------------------
- id                      text PK (UUID)
- organizationId          FK → organization
- userId                  FK → user  (the admin or management user who owns this account)
- accountType             enum: CANTEEN_ADMIN | MANAGEMENT
- label                   text  (user-friendly name, e.g. "Main SBI Account")
- method                  enum: BANK_ACCOUNT | UPI
- bankAccountNumber       text, AES-256-GCM encrypted, nullable  (BANK_ACCOUNT only)
- bankIfsc                text, nullable                          (BANK_ACCOUNT only)
- bankAccountHolderName   text, nullable                          (BANK_ACCOUNT only)
- upiVpa                  text, nullable                          (UPI only)
- razorpayContactId       text, nullable  (auto-created via RazorpayX Contact API)
- razorpayFundAccountId   text, nullable  (auto-created via RazorpayX Fund Account API)
- status                  enum: ACTIVE | BLOCKED | PENDING_VERIFICATION
- blockedByUserId         FK → user, nullable
- blockedAt               timestamp, nullable
- blockReason             text, nullable
- createdAt               timestamp
- updatedAt               timestamp
- Unique constraint on (organizationId, userId, method + account details) to prevent duplicates

1.2  New table: canteen_payment_routing
-----------------------------------------
- id                  text PK (UUID)
- canteenId           FK → canteen, UNIQUE  (one routing config per canteen)
- settlementAccountId FK → settlement_account
- overriddenByUserId  FK → user, nullable  (set when management overrides default)
- overriddenAt        timestamp, nullable
- createdAt           timestamp
- updatedAt           timestamp
- Default routing resolves to the admin's own account via their KIOSK device assignment.

1.3  New table: settlement_ledger
-----------------------------------
Tracks settlement owed and paid for every eligible canteen order.
- id                  text PK (UUID)
- organizationId      FK → organization
- settlementAccountId FK → settlement_account
- orderId             FK → order, nullable
- grossAmount         doublePrecision   (full order total including fee)
- platformFee         doublePrecision   (2% of subtotal)
- netAmount           doublePrecision   (grossAmount - platformFee)
- entryType           enum: DEBIT | REVERSAL  (REVERSAL for cancelled-PAID orders refunded)
- status              enum: PENDING | PROCESSING | SETTLED | FAILED
- razorpayPayoutId    text, nullable
- settledAt           timestamp, nullable
- failureReason       text, nullable
- createdAt           timestamp

1.4  New table: settlement_batch
----------------------------------
Groups ledger entries per account into a single daily RazorpayX payout.
- id                  text PK (UUID)
- organizationId      FK → organization
- settlementAccountId FK → settlement_account
- totalGross          doublePrecision
- totalFee            doublePrecision
- totalNet            doublePrecision
- orderCount          integer
- status              enum: PENDING | PROCESSING | SETTLED | FAILED | PARTIALLY_FAILED
- razorpayPayoutId    text, nullable
- processedAt         timestamp, nullable
- failureReason       text, nullable
- createdAt           timestamp

1.5  Modify existing table: order
------------------------------------
Add column:
- platformFee   doublePrecision, default 0  (the 2% fee stored on the order row)

=======================================================
PHASE 4: SETTLEMENT LEDGER & PAYOUT ENGINE
=======================================================

4.1  Modify: app/api/payments/verify/route.ts
  After marking order PAID:
  - Resolve routing: order.canteenId → canteen_payment_routing → settlement_account
  - If no routing configured: create ledger entry marked PENDING against a null/platform account
    (funds stay in platform Razorpay; flagged in management dashboard as "unrouted")
  - Create settlement_ledger entry (type: DEBIT, status: PENDING)

4.2  Modify: app/api/payments/wallet-fallback/route.ts
  - Same ledger creation logic after successful wallet debit

4.3  Modify: app/api/admin/orders/[id]/status/route.ts (or equivalent cancellation path)
  - When a PAID order is CANCELLED and refunded: create settlement_ledger entry (type: REVERSAL)
  - Reversal deducted from next settlement batch for that account

4.4  NEW: lib/razorpay-payout.ts
  RazorpayX utility functions:
  - createContact(name, email, phone)  → razorpayContactId
  - createFundAccount(contactId, bankDetails | upiVpa)  → razorpayFundAccountId
  - createPayout(fundAccountId, amountPaise, reference)  → razorpayPayoutId
  - getPayoutStatus(payoutId)  → status string
  Uses env vars: RAZORPAY_PAYOUT_KEY_ID, RAZORPAY_PAYOUT_KEY_SECRET (RazorpayX credentials)

4.5  NEW: app/api/settlements/process/route.ts
  Daily CRON / manually triggered endpoint (protected by CRON_SECRET header):
  - Query all PENDING ledger entries grouped by settlementAccountId
  - For each account:
    * Sum netAmounts (DEBITs minus REVERSALs)
    * Skip if account is BLOCKED or net amount ≤ 0
    * Create settlement_batch row
    * Call lib/razorpay-payout.ts → createPayout()
    * Update ledger entries to PROCESSING
  - Returns summary of batches initiated

4.6  NEW: app/api/webhooks/razorpay-payout/route.ts
  Receives RazorpayX payout webhook callbacks:
  - Verify webhook signature (RAZORPAY_PAYOUT_WEBHOOK_SECRET)
  - On payout.processed → update batch + ledger entries to SETTLED
  - On payout.failed    → update batch + ledger entries to FAILED, store failureReason
  - On payout.reversed  → mark FAILED, trigger re-queue logic

=======================================================
KEY DECISIONS
=======================================================

- 2% platform fee is VISIBLE to parents as "Platform Fee" line item at checkout
- All inbound payments flow through the single platform Razorpay account
- Outbound settlements use RazorpayX Payout API (daily automated batches)
- Bank account numbers encrypted at rest with AES-256-GCM (same pattern as Razorpay key secrets)
- Admin→canteen default routing resolved via existing device assignment system
- Management can add accounts on behalf of admins but admins can also self-register
- Management has full block/unblock authority and can override payment routing per canteen
- CASH orders are excluded from the settlement ledger (money is physically at the canteen)
- Existing organizationPaymentConfig table is UNTOUCHED (handles inbound keys, not outbound settlement)
- Blocked account fallback: funds stay in platform Razorpay, flagged as "unrouted" in dashboard
- Refund reversal: cancelled PAID orders create a REVERSAL ledger entry deducted from next batch


=======================================================
OPEN QUESTIONS / FURTHER CONSIDERATIONS
=======================================================

1. IFSC validation: regex ^[A-Z]{4}0[A-Z0-9]{6}$ for now; full RBI list API validation later.

2. CASH / UPI-direct orders at kiosk: excluded from settlement ledger since cash is already
   physically at the canteen. Only ONLINE and WALLET payment methods go through ledger.

3. Refund flow: when a PAID order is cancelled, wallet is refunded. A REVERSAL ledger entry
   is created against the same settlement account. Net of DEBIT + REVERSAL is settled in next batch.

4. Unrouted canteens: if a canteen has no settlement_account configured, orders still succeed
   (to avoid blocking parents from buying food). Funds accumulate in platform Razorpay and are
   flagged prominently in the management settlement dashboard with a warning banner.

5. RazorpayX account activation: RazorpayX (for payouts) is a separate product from the
   standard Razorpay payment gateway and requires a separate KYC/activation with Razorpay.
   The app will gracefully degrade if payout credentials are not yet configured — ledger entries
   are created but batching is a no-op until env vars are present.
