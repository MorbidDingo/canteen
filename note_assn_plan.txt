Plan: Assignments & Notes System
TL;DR — Build a classroom content system. Management grants posting permission to selected users via a dedicated permissions table. Permitted users create Assignments (with due dates) or Notes (reference material). Both support file attachments and targeted audiences (whole org, class, section, or individual). Students/parents submit work on assignments privately. Notes live in a Library-adjacent experience; both surface in a new "Assignments" section in the parent/general bottom nav.

Phase 1 — Database Schema & Migration
Steps

Add 6 new tables to schema.ts:
content_permission — tracks who is permitted to post, scoped to ASSIGNMENT | NOTE | BOTH, granted by MANAGEMENT
content_post — unified row for both types: type, title, body (markdown), dueAt (nullable), status: DRAFT | PUBLISHED | CLOSED
content_post_attachment — files on a post; storageBackend: S3 | CLOUDINARY, storageKey, mimeType, size
content_post_audience — rows per target: audienceType: ALL_ORG | CLASS | SECTION | USER | GROUP, optional className, section, userId, groupId
content_submission — one row per submitter per post; status: SUBMITTED | RESUBMITTED; holds optional textContent
content_submission_attachment — files in a submission; same storage fields as post attachments

Add 4 new tables for tags & groups:
content_tag — org-scoped tag (e.g. "Math", "Science", "HR Updates"); fields: id, organizationId, name, color (nullable hex), createdBy, createdAt
content_post_tag — junction: postId + tagId; unique on (postId, tagId)
content_group — named user group within an org (e.g. "Teaching Staff", "Grade 5 Teachers"); fields: id, organizationId, name, description (nullable), createdBy, createdAt, updatedAt
content_group_member — membership: groupId + userId; unique on (groupId, userId); createdAt

Modify content_post_audience to support GROUP targeting:
  - Add GROUP to audienceType enum: ALL_ORG | CLASS | SECTION | USER | GROUP
  - Add groupId FK (nullable) referencing content_group
  - Update CHECK constraint: GROUP requires groupId IS NOT NULL, all others NULL
  - Update unique constraint to include groupId

Add Drizzle relations for all new tables
Generate migration drizzle/00XX_assignments_notes.sql
Relevant files: schema.ts (end of file, after readableBook tables)

Phase 7 — AI Integration with Content System (depends on Phases 3, 4, 6; Certe+ gated)

Overview
Extend the existing Claude-powered AI assistant (lib/ai/) to understand and interact with assignments & notes. Add a pgvector-backed RAG pipeline so the AI can answer questions about uploaded content (PDF, Word, Excel, TXT) scoped to the user's audience access. The AI can also create assignments and notes on behalf of permitted users. Everything AI is gated behind Certe+ subscription, consistent with the canteen AI assistant.

Sub-phase 7A — pgvector Infrastructure & Document Ingestion Pipeline
Steps

Enable pgvector extension:
  Migration: CREATE EXTENSION IF NOT EXISTS vector;
  Add to drizzle config if needed

New tables in schema.ts:
  content_document_chunk — id, postId FK (content_post), attachmentId FK (content_post_attachment), chunkIndex (int), content (text), embedding (vector(1536)), metadata (jsonb: page number, section, original filename), organizationId FK, createdAt
  Index: HNSW index on embedding column using vector_cosine_ops for fast ANN search
  Index: (organizationId, postId) for scoped filtering

Document processing pipeline — lib/ai/content-embeddings.ts:
  Supported MIME types: text/plain, application/pdf, application/vnd.openxmlformats-officedocument.wordprocessingml.document (.docx), application/vnd.openxmlformats-officedocument.spreadsheetml.sheet (.xlsx), application/msword (.doc), application/vnd.ms-excel (.xls)
  Processing flow:
    1. On attachment upload (POST /api/content/posts/[id]/attachments), if MIME type is supported, enqueue processing
    2. Fetch file from S3/Cloudinary
    3. Extract text: pdf-parse for PDF, mammoth for .docx, xlsx for .xlsx, raw read for .txt
    4. Chunk text with overlap: 512 tokens per chunk, 64 token overlap (use tiktoken for accurate token counting)
    5. Generate embeddings via Anthropic's voyager or OpenAI text-embedding-3-small (whichever is configured; default OpenAI for cost)
    6. Insert chunks + embeddings into content_document_chunk
  Background processing: use a simple async queue (process after response); no external job queue needed for MVP
  Re-processing: on attachment delete, CASCADE removes chunks; on re-upload, old chunks auto-removed

New packages:
  pgvector (drizzle pgvector support)
  pdf-parse — PDF text extraction
  mammoth — .docx text extraction
  xlsx — Excel text extraction
  openai — for text-embedding-3-small (embeddings only; chat stays Anthropic)
  tiktoken — accurate token counting for chunking

Migration drizzle/00XX_content_vector_search.sql:
  CREATE EXTENSION IF NOT EXISTS vector;
  CREATE TABLE content_document_chunk with vector(1536) column
  CREATE INDEX using hnsw (embedding vector_cosine_ops) WITH (m = 16, ef_construction = 64)
  Standard indexes on (organization_id, post_id), (attachment_id)

Sub-phase 7B — Audience-Scoped Vector Search
Steps

Search function — lib/ai/content-search.ts:
  queryContentChunks(orgId, userId, query, options?):
    1. Generate embedding for the query string
    2. Resolve user's accessible postIds:
       - If user is an author: include their own posts
       - For reader access: run same audience resolution as feed API (children's class/section, group memberships, direct user targeting, ALL_ORG)
       - Never return chunks from posts the user cannot access
    3. Execute vector similarity search:
       SELECT c.*, 1 - (c.embedding <=> $queryEmbedding) AS similarity
       FROM content_document_chunk c
       WHERE c.organization_id = $orgId
         AND c.post_id = ANY($accessiblePostIds)
       ORDER BY c.embedding <=> $queryEmbedding
       LIMIT $topK
    4. Filter by similarity threshold (> 0.7 default)
    5. Return chunks with post title, type, author name for citation
  Options: topK (default 5), similarityThreshold (default 0.7), type filter (ASSIGNMENT | NOTE | both)

Reuse audience resolution from lib/content-audience.ts — extract shared helper getAccessiblePostIds(orgId, userId) that both the feed API and vector search can call

Decisions
Embedding model: OpenAI text-embedding-3-small (1536 dimensions) — cheaper than Anthropic for bulk embeddings; chat stays on Claude
Vector index: HNSW with cosine similarity — best balance of speed and recall for <100K chunks per org
Chunk strategy: 512 tokens with 64 overlap — good for mixed content (assignments tend to be short, notes can be long)
AI-created content: always DRAFT, never auto-published — safety guardrail
File upload via AI: explicitly blocked — users must use the upload UI; AI can only create text content
Audience scoping in search: mandatory, not optional — the vector search MUST filter by accessible posts; this is a security boundary, not a feature flag

Decisions
Posting permission: dedicated content_permission table (MANAGEMENT-granted)
Audience: child.className + child.section for parents/students; content_group for GENERAL accounts and cross-cutting teams (no new class-management table)
Subject/Topic tagging: content_tag + content_post_tag junction; org-scoped; created by management or permitted users; used for filtering, not access control
User grouping: content_group + content_group_member; management or permitted users create groups; GROUP is a first-class audienceType alongside ALL_ORG/CLASS/SECTION/USER
Storage: S3 for documents/video, Cloudinary for images
Due dates: dueAt timestamp; submissions blocked after it passes
UI placement: separate "Assignments" section in parent/general bottom nav
